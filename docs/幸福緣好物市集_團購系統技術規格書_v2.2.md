### 1.3 零欄位異動原則（v2.2 核心約束）

```
以下行為在本系統中被明確禁止，屬於架構違規：

❌ 對既有 9 張表新增任何欄位（無論 append-only 還是插入中間）
❌ 修改既有 COL 常數物件中任何已定義的 key 或 index 值
❌ 在任何既有 GAS function 中加入感知新欄位的判斷分支
❌ 讓一般零售流程的帳務（商品帳本）與團購分潤（團購帳務表）產生交叉寫入

✅ 合法的做法：
   - 所有團購資料寫入第二章定義的 5 張新表
   - 新 GAS function 透過 order_id / phone 等已存在的鍵值與既有表關聯
   - 一般零售流程完全不感知團購模組的存在
```

### 1.4 明確聲明：既有養清倉管主流程不得被破壞


## 二、新增表設計（共 5 張全新工作表）

## 三、LIFF ID Token 後端驗證流程

### 3.1 完整驗證流程

```
步驟1：客人在LINE內開啟LIFF頁面（會員專區、團長專區、集單登記頁面等）
步驟2：前端呼叫 liff.getIDToken() 取得 ID Token
       （不使用 liff.getProfile() 的結果作為身份依據，因為那是前端自陳，未經驗證）
步驟3：前端呼叫後端 GAS API 時，將 ID Token 作為參數傳送
       （前端不得自行組裝或傳送 line_user_id 欄位給後端）
步驟4：後端 GAS 收到 ID Token 後，呼叫 LINE 官方驗證端點：
       POST https://api.line.me/oauth2/v2.1/verify
       參數：id_token（前端傳來的Token）、client_id（此LIFF對應的Channel ID）
步驟5：LINE 回傳驗證結果，包含欄位 sub（即真正的 LINE User ID）與 exp（有效期限）
步驟6：後端檢查回傳結果是否驗證成功、Token是否在有效期限內
步驟7：驗證通過後，後端才使用這組「驗證過的」User ID，去查詢/寫入
       團主來源表、團購明細表等資料
       （注意：LINE User ID 一律儲存於新表，不寫入既有9張表）
```

### 3.2 不信任前端 line_user_id 的具體規則

```
規則：任何 API 端點，一律不得將前端傳來的「line_user_id」參數直接當作
      查詢或寫入資料的依據，唯一合法的 User ID 來源是步驟4-6驗證後取得的 sub 值

理由：若接受前端直接傳參數，任何人可透過瀏覽器開發工具或攔截封包工具，
      偽造 line_user_id 參數，冒充其他人查詢分潤金額、訂單狀態等隱私資料
```

### 3.3 驗證失敗處理

```
若驗證失敗（Token過期、Token不合法、client_id不符、LINE驗證端點回傳錯誤）：
  → 後端拒絕該次請求，回傳明確錯誤訊息（例如「登入狀態已失效，請重新整理頁面」）
  → 前端顯示錯誤提示，引導客人重新從LINE OA選單或聊天視窗進入
    （重新觸發LIFF初始化以取得新的、有效的ID Token）
  → 嚴禁在驗證失敗時，退而使用前端傳來的電話號碼或其他參數直接查詢資料，
    必須要求重新完成驗證流程
```

## 四、所有寫入點的 LockService 加鎖清單

以下每一個寫入點，涉及「先讀取現有數值 → 判斷或計算 → 寫回」的操作，
一律使用 `LockService.getScriptLock()` 包住整個讀寫區塊，避免並發時互相覆蓋：

```
清單1：團購明細表寫入新登記時，更新該活動的累積數量
      （讀取現有累積值 → 加上新登記數量 → 寫回団購活動表或即時計算欄位）

清單2：客戶訂單成立時，更新對應商品的「預留庫存」數量
      （讀取現有預留值 → 加上這筆訂單數量 → 寫回）

清單3：訂單狀態改為「已完成」時，將預留庫存轉為正式扣減
      （讀取預留值與實際庫存值 → 計算後寫回實際庫存，並將預留值歸零）

清單4：訂單狀態改為「取消」或「退貨」時，釋放預留庫存
      （讀取預留值 → 扣除該筆訂單數量 → 寫回預留值）

清單5：團長確認下單，將團購明細表的多筆有效登記批次轉成客戶訂單表的多筆訂單
      （需鎖定該活動ID，避免同一活動被重複觸發兩次「確認下單」動作）

清單6：團主來源表寫入LINE User ID綁定時
      （讀取是否已有其他記錄使用同一組User ID或同一組電話 → 判斷後寫入，
        避免同時間兩筆綁定請求造成資料衝突；
        此清單改寫「団主來源表」，不寫既有「會員」表）

清單7：未取貨次數累加與團購資格狀態判斷
      （讀取現有未取貨次數 → 加1 → 判斷是否達3次 → 寫回次數與資格狀態，
        避免同一會員的多筆逾期訂單同時觸發判斷造成次數算錯）
```

加鎖原則：每個清單項目的鎖定範圍應盡量精簡（例如鎖定單一活動ID或單一商品ID的範圍，
而非整份表鎖定），避免不必要地拖慢其他不相關操作的執行速度；但寧可鎖定範圍稍大也不可遺漏
任何一個讀寫點。

## 五、底價快照規則

### 5.1 底價的兩種來源

```
簡單開團模式（MVP）：底價來自「團購商品設定表」的固定 base_price 欄位
集單模式：底價來自商品設定中，依實際達成數量對應的階梯底價
          （階梯設定儲存於団購商品設定表或另行設計的附屬結構，待集單模式啟動時細化）
```

### 5.2 快照時機與寫入規則

```
時機：團長確認成團、系統批次建立客戶訂單的當下
規則：將當時適用的底價（依上述兩種來源判斷後）複製一份數值，
      寫入「団購活動表.base_price_snapshot」欄位
效果：之後不管「団購商品設定表」的底價如何調整，
      這次成團的所有訂單永遠依照成團當時鎖定的快照數值結算，
      不會被事後的價格變動影響
注意：base_price_snapshot 欄位位於「団購活動表」（非客戶訂單表），
      對應關係為：
        団購明細表.campaign_id → 団購活動表.id → 団購活動表.base_price_snapshot
```

### 5.3 利潤計算依賴快照，不依賴即時查詢

```
供應方應得金額 = base_price_snapshot × 數量（逐項計算後加總）
團長應得金額 = Σ（(開團售價 − base_price_snapshot) × 數量） − 理貨代收費
```

任何結算、月結報表、團長專區查詢，計算依據一律是「団購活動表」中已寫入的
`base_price_snapshot` 欄位，不得重新查詢「団購商品設定表」的「目前」底價。

## 六、沖銷與帳務追溯規則

### 6.1 沖銷觸發時機

```
訂單狀態變更為「取消」或「退貨」時觸發：
- 若該訂單尚未產生任何「団購帳務表」記錄（例如客人尚未取貨、根本還沒結算過），
  則不需要沖銷，直接將訂單標記為取消/退貨即可
- 若該訂單已經產生過分潤記錄（已完成後才發現要退），才需要寫入沖銷記錄
```

### 6.2 沖銷記錄的寫入方式

```
針對原始記錄的訂單ID，寫入對應的沖銷記錄：
  訂單ID：與原始記錄相同
  角色：與原始記錄相同（供應方/團長/幸福緣，逐筆對應）
  金額：與原始記錄相等，但正負號相反
  狀態：標記為「沖銷」
不得直接刪除原始記錄，確保異動軌跡完整可查
```

### 6.3 帳務可追溯原則

```
任何一筆金額，都必須能夠回答以下三個問題：
1. 這筆錢屬於哪一筆訂單（訂單ID）
2. 這筆錢屬於誰（角色、對象）
3. 這筆錢是怎麼計算出來的（對應的商品、數量、單價/底價快照）

月結報表、團長專區、供應方對帳單，皆從「団購帳務表」篩選＋加總產生，
篩選時必須將「正常」與「沖銷」狀態的記錄一併納入計算（沖銷記錄會自動抵銷對應的正常記錄），
不得只呈現「正常」狀態記錄而忽略沖銷記錄，否則會高估應付金額
```

## 七、庫存預留與釋放規則

### 7.1 三個時點

```
時點1（訂單成立）：
  客戶訂單寫入，狀態＝「待取貨」
  → 對應商品的預留庫存數量增加該筆訂單的數量
  → 實際庫存數字不變動

時點2（取貨完成）：
  訂單狀態由「待取貨」改為「已完成」
  → 該筆訂單的預留數量轉為正式扣減
  → 實際庫存數字在此刻才真正減少
  → 預留庫存數量歸零（該筆訂單部分）

時點3（取消或退貨）：
  訂單狀態改為「取消」或「退貨」
  → 該筆訂單的預留數量歸零釋放，回到可販售庫存
  → 若訂單先前已進入「已完成」狀態才轉為退貨，需視實際退貨狀況判斷
    是否將貨品實際入庫（回補實際庫存），此判斷由管理員人工於後台操作確認
```

### 7.2 可販售庫存計算公式

```
可販售庫存 = 商品主檔實際庫存 − 所有「待取貨」狀態訂單的預留數量總和
```

### 7.3 超賣防範

```
訂單成立時，系統應檢查「可販售庫存」是否足夠這筆訂單的數量：
- 足夠：正常成立，進入時點1的預留流程
- 不足：後台應提示警告，是否強制阻擋依商品性質由管理員決定
此檢查與預留數量的讀寫，皆須包在第四章清單2的LockService鎖定範圍內，
確保多個團長同時賣同一商品時，數字不會因並發而算錯
```

## 八、團主／團員 UI 流程

### 8.1 團主（團長）開團流程

```
第一階段（MVP）：
  團長私下告知管理員想開的商品、固定加價、期限
  → 管理員在後台建立「団購活動表」記錄
  → 系統產生專屬集單連結（例如 ?campaign=A001）
  → 團長自行分享推廣，自己盯進度，自己決定何時確認下單

第二階段（進階，待第一階段驗證跑順後啟動）：
  團長透過LIFF自助登入
  → ID Token驗證後，確認「団主來源表.member_level」為「團長」
  → 自行挑選已標記「適用集單模式」的商品
  → 設定固定加價、門檻類型、截止時間
  → 送出建立，系統自動產生活動ID與集單連結
```

### 8.2 團員（客人）下單流程

```
客人點擊團長分享連結，或訂購頁面「團購中」分類頁籤的活動卡片
→ 進入LIFF集單頁面，顯示商品資訊、階梯進度、剩餘時間
→ 頁面自動觸發LIFF取得ID Token（客人無感，不需手動操作）
→ 客人用「+/−」按鈕輸入登記數量（非自由輸入框，避免亂填）
→ 送出登記
→ 後端驗證ID Token → 確認為合法使用者 → 寫入団購明細表（第四章清單1鎖定範圍內）
→ 回傳登記成功確認，並觸發LINE推播通知（登記成功、進度更新等情境）
```

### 8.3 團長專區（自助查詢）

```
團長點擊「團長專區」入口
→ LIFF自動取得ID Token → 後端驗證 → 查詢「団主來源表.line_uid」確認為「團長」
→ 是團長：顯示其自己開過的所有活動列表（進行中／已結束）、
  每團底下每筆訂單的取貨狀態、累積分潤金額（可依月份篩選，
  數字來源皆為「団購帳務表」，見第六章追溯原則）
→ 不是團長：顯示提示訊息，導回一般會員專區，不開放進入
→ 看不到：底價、底價快照、供應方名稱、其他團長的資料
  （隱私邊界與既有規則一致，不因自助查詢功能而放寬）
```

### 8.4 會員專區（一般客人／首次辨識）

```
客人點擊「會員專區」入口
→ LIFF自動取得ID Token → 後端驗證取得真正的LINE User ID
→ 查詢「団主來源表」：此 line_uid 是否已有紀錄
  → 有記錄：直接進入會員專區，顯示訂單、點數、優惠，不需再輸入手機號碼
  → 無記錄：顯示「首次填寫資料」表單
    → 若輸入電話號碼在「會員」表已有既有資料：
      比對後在「団主來源表」新增一筆（member_level=一般會員，line_uid 填入驗證 sub 值），
      不需重填姓名等其他既有資料，也不修改「會員」表任何欄位
      （此寫入動作納入第四章清單6鎖定範圍）
    → 若電話號碼在「會員」表也查無資料：視為全新會員，需完整填寫姓名、電話，
      在「會員」表建立新會員記錄（既有流程，不修改），
      同時在「団主來源表」建立一筆 line_uid 綁定記錄
→ 綁定完成後，直接導向會員專區，不需再次操作

注意：整個 8.4 流程唯一寫入既有「會員」表的情境，
      是「全新會員建立」這個既有流程本來就會做的事，行為不變。
      LINE User ID 綁定一律只寫「団主來源表」，不修改既有會員欄位。
```

### 8.5 後台查單與匯出統計

```
後台查單：
  以電話號碼查詢客人所有「待取貨」訂單（可能橫跨不同團長的不同活動）
  → 勾選欲結清的訂單（可複選）→ 逐筆確認商品/數量/收款方式/金額 → 填寫備註
  → 確認取貨 → 系統對每筆勾選的訂單各自獨立觸發：
    第五章底價快照結算、第二章2.5団購帳務表寫入、第七章時點2庫存扣減、
    訂單狀態改為「已完成」（多筆訂單各自獨立處理，不合併計算）

匯出統計：
  月結報表：依「角色＋對象」篩選「団購帳務表」，產出供應方／團長／幸福緣
    各自的逐筆明細單（日期、訂單ID、商品、數量、單價、小計、加總），可匯出/列印
  團長專區查詢：即時篩選「団購帳務表」中「角色＝團長且對象＝該團長」的記錄，
    供團長自助查詢，數字與月結報表一致
```

## 九、v2.2 與 v2.1 差異說明

本節為版本升級的正式記錄，明確說明 v2.2 取代 v2.1 的原因與範圍。

### 9.1 核心架構差異

| 項目 | v2.1（舊） | v2.2（新） |
|------|-----------|-----------|
| 既有表欄位 | 允許 append-only 新增欄位 | **既有 9 張表零欄位異動** |
| PRODUCTS 新增欄位 | 新增 5 欄（supplier/base_price 等） | **取消，改放「团購商品設定表」** |
| STOCK_LOG 新增欄位 | 新增 2 欄（unit_price/subtotal） | **取消，团購進出貨資訊透過訂單關聯推算** |
| ORDERS 新增欄位 | 新增 4 欄（group_leader/group_price 等） | **取消，改放「団購活動表」與「団購明細表」** |
| MEMBERS 新增欄位 | 新增 1 欄（line_user_id） | **取消，LINE UID 改放「団主來源表」** |
| 底價快照儲存位置 | 客戶訂單表.底價快照欄位 | **団購活動表.base_price_snapshot** |
| LINE UID 儲存位置 | 會員.line_user_id | **団主來源表.line_uid**（一般會員與團長均存此處） |
| 商品帳本 | 規劃新增「代收代寄服務」選項值 | **商品帳本完全不涉及团購，維持現狀** |

### 9.2 5 張新表名稱異動

| v2.1 表名 | v2.2 表名 | 說明 |
|-----------|-----------|------|
| 團購商品表（階梯底價） | 團購商品設定表 | 合併商品設定與底價設定，含 supplier/base_price 等原本規劃放入 PRODUCTS 的欄位 |
