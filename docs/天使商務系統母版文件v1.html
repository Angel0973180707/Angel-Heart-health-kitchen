01

系統簡介與設計理念

什麼是天使商務系統？

天使商務系統（Angel Commerce System，ACS）是為小型商家、個人品牌、在家接單創業者、教練、教師、課程講師、療癒工作者、手作品牌與活動主辦者設計的一站式商務管理母版。以 Google Sheets 為資料庫，Google Apps Script（GAS）為後端，搭配 GitHub Pages 前端與 LINE 即時通知，讓老闆不需要購買昂貴的系統，即可擁有完整的商務管理能力。第一版參考實作為養清倉管系統，後續母版化時，將支援商品銷售、課程報名、國內營隊、旅療活動與國際旅遊性質活動。

本母版文件記錄 ACS 的核心規格，以養清倉管系統（Angel-Heart-health-kitchen）為第一個參考實作案例。未來部署給其他客戶時，應從獨立的 Template Repo 建立，不沿用正式營運 repo。

06

部署步驟（新客戶安裝流程）

⚠ 草案注意：目前 Template Repo 尚未從正式營運 repo 獨立。本章流程為目標狀態描述，部分步驟（尤其步驟 6）在正式 Template Repo 建立前仍屬規劃。

事前準備清單

項目說明
Google 帳號客戶自己的帳號，不可用供應商帳號
LINE OA 帳號申請免費 LINE Official Account，取得 Channel Access Token
GitHub 帳號用於 fork Template Repo 並啟用 GitHub Pages
LINE User ID老闆本人的 LINE UID（可用 LINE OA 後台查詢）

安裝步驟

1
複製 Google Spreadsheet 以母版試算表為範本，「建立副本」到客戶的 Google 帳號，取得新的 Spreadsheet ID。
2
建立 Google Apps Script 專案 在試算表中開啟「擴充功能 → Apps Script」，貼上 GAS 程式碼。現階段需更新 gas_event_clean.js 內 SPREADSHEET_ID 常數；母版化目標是改由 Script Property SPREADSHEET_ID 管理。
3
執行初始化函式 在 GAS 編輯器依序執行：
① adminSetupSchema()　← 補齊所有工作表欄位
② adminSetupPhoneFormats()　← 電話欄位設純文字格式
③ installTriggers()　← 安裝每日低庫存 / 月結定時觸發器
4
設定 Script Properties 在 GAS「專案設定 → 指令碼屬性」填入所有機密參數（見第 5 章）。
5
部署為 Web App 「部署 → 新增部署作業 → 類型：Web 應用程式」，執行身份：「我」，存取：「所有人」，取得 Web App URL。
6
Fork Template Repo（草案：尚未建立） 未來應建立獨立的 Template Repo（暫訂名稱：Angel-Commerce-System-Template），與正式營運的 Angel-Heart-health-kitchen repo 完全分離。
客戶 fork Template Repo（不是正式營運 repo）→ 更新所有頁面中的 GAS_URL → 啟用 GitHub Pages。
⚠ 在 Template Repo 正式建立前，不應直接讓客戶 fork 正式營運 repo。
7
啟用 GitHub Pages 在 fork 的 repo 設定中，開啟 Pages → Source: Deploy from a branch（main）。等約 1 分鐘，前端即上線。
8
功能驗收 執行 GAS 中的 adminTestOldCodeCompatibility() 確認欄位索引正確；執行 runTestEventRegistrationV26() 確認活動報名模組通過（預期 pass=14 fail=0）。

日常操作流程

商品銷售流程

顧客下單（shop.html）→ 老闆收 LINE 通知 → 後台確認訂單（扣庫存）→ 出貨 → 確認付款（自動加點）→ 完成

活動報名流程

顧客報名（event.html）→ 老闆收 LINE 通知 → 後台審核（通過 / 未錄取）→ 通知繳費 → 後台確認付款 → 寫入活動帳本

若需退款：後台「退款取消」→ 系統自動寫退款帳目 + 名額歸還

退貨流程（基礎版，待審查）

後台建立退貨申請 → 確認退貨（三步驟：庫存回補 / 帳本記錄 / 點數扣回）→ 完成

⚠️ 若步驟執行失敗會標記 needs_review，需人工確認後手動補執行。退貨模組尚待獨立審查，未納入 v1.0 草案封版範圍。

可選金流模組規劃

模組定位

  • 金流是客戶可選模組，不是所有客戶必開。預設情況下，所有客戶以「手動匯款 + 後台人工確認付款」為穩定基礎，不需串接第三方金流即可正常營運。
  • 第一版仍以手動收款版為核心，第三方金流版列為可加購模組，按需開通。
  • 第三方金流需另開獨立安全審查任務，不可與商品分類管理、運費設定、商品編輯等一般功能任務混入同一個 PR 或部署作業。
⚠ 金流模組安全審查為獨立作業。未完成審查前，不得將金流相關程式碼合併入正式分支或對客戶部署。

金流分級

等級名稱收款方式後台確認適合對象
Lv 0 手動收款版 ATM 轉帳 / 現金 管理員手動確認，填入末五碼或實收金額 剛起步、訂單量少、可接受人工對帳的小商家
Lv 1 半自動對帳版 ATM 轉帳為主 顧客填末五碼 + 金額 + 時間 → 系統比對疑似付款 → 管理員一鍵確認 訂單量中等、希望減少手動核帳工時的商家
Lv 2 第三方金流版 信用卡 / ATM / 超商代碼 付款成功後系統自動確認，無需管理員手動操作 有一定訂單量、需要自動化收款的商家；需向金流商申請特店帳號（綠界 ECPay / 藍新 NewebPay / LINE Pay）
Lv 3 活動報名金流版 同 Lv 2，但流程不同 報名審核通過後才產生付款連結；不同票種 / 住宿方案對應不同金額；付款成功後更新活動報名狀態並寫入活動帳本 需要「先審核後收款」的活動、旅遊、課程主辦單位

共用付款核心流程

1
建立訂單 / 報名記錄 顧客完成選購或報名填表 → GAS 寫入訂單 / 報名資料，依模組進入待確認或待付款狀態；商品訂購可先待確認，活動報名可先申請中或待付款。
2
根據 payment_provider 判斷收款模式 manual → 顯示匯款帳號與末五碼回填說明
ecpay / newebpay / linepay → 呼叫金流 API 產生付款連結 → 導向金流頁面
3
顧客完成付款 手動收款：顧客完成匯款後填末五碼回填表單
第三方金流：顧客在金流頁面完成刷卡 / 轉帳 / 超商繳款
4
付款通知回傳（callback） 第三方金流：金流商 POST 通知至 GAS doPost Web App URL
手動收款：管理員在後台手動觸發確認，或系統自動比對末五碼
5
驗簽與冪等檢查 第三方金流：GAS 驗證簽章（SHA256）確認通知來源合法，驗簽失敗一律拒絕
冪等檢查：若訂單 / 報名已是已付款狀態，跳過後續步驟，防止重複處理
6
更新訂單 / 報名狀態 商品訂單 status → 已付款；活動報名 fee_status → 已繳費
7
寫入商品帳本 / 活動帳本 依 auto_write_account 設定決定是否自動寫入；商品訂購與活動報名帳本仍需分流，不得混寫同一筆帳目
8
加會員點數 依 auto_add_points 與 points_earn_rate 計算；運費不計入點數累積基數
9
回傳付款結果 + LINE 推播 前端顯示 payment_success_text 或 payment_failed_text；推播 LINE 通知老闆（新訂單確認）與顧客（若有綁定 LINE）

安全規則

⚠ 以下六條規則為強制要求。任何金流等級的實作必須全部遵守，不得以「趕進度」或「先上線再說」為由省略任何一條。
規則說明
不信任前端金額 付款金額必須由後端（GAS）依訂單主檔重新計算，前端傳入的金額僅供顯示用，不可直接用於帳務寫入或點數計算
callback 必須驗簽 收到第三方金流 callback 時，必須用 HashKey / HashIV 驗證簽章，驗簽失敗一律拒絕，並記錄異常日誌
callback 必須防重複處理 同一筆訂單收到多次 callback（金流商重試機制）時，必須做冪等判斷：確認已付款則跳過，不得重複寫入帳本或重複加點數
金流密鑰不可寫死在前端 HashKey、HashIV、merchant_id 等機密僅存於 GAS Script Properties，不可出現在任何前端 HTML / JS 檔案、GitHub repo 或本文件
退款流程需獨立審查 退款涉及金流商 API 逆向呼叫與帳務逆向操作，必須與正向付款流程分開審查，不得在同一任務中實作，需另立退款安全審查任務
商品帳本與活動帳本仍需分流 即使引入自動金流，商品訂購收入仍需寫入商品帳本，活動報名費仍需寫入活動帳本，不得合流或共用同一筆帳目記錄

商品型態與客製化商品規劃

⚠ 本章為規劃草案。P1 僅實作現貨商品編輯；P2 / P3 / P4 尚未實作，不列入 v1.0 草案封版範圍。

1. sale_type 銷售型態設計

商品主檔新增 sale_type 欄位，以字串識別銷售型態，不使用 boolean。字串值可隨業務擴充，boolean 無法應對三種以上型態。

值說明
normal一般現貨，依庫存 qty 銷售
preorder預購，不扣現貨 qty，計 preorder_count
made_to_order接單製作 / 預約訂製，客人填自由需求，後台人工確認或報價
menu_select點選菜單，商家預建選項群組，客人點選組合，系統計算金額
hidden不公開上架，可建立但不顯示於前台
sale_type 與 status 的分工：
status（上架 / 下架 / 停用）負責商品的可見性與是否可銷售。
sale_type 負責銷售型態，決定前台顯示邏輯與訂單流程。
兩者正交，不可混用。例如：一個「預購商品」可以同時處於「下架」狀態（status=下架，sale_type=preorder），代表預購已截止或暫停。

2. 預購商品欄位（sale_type = preorder）

欄位說明
preorder_note預購說明，前台顯示給客人（如：預計 2 週後到貨）
preorder_deadline預購截止日，超過後前台自動顯示「預購已截止」

⚠ 以下規則為強制要求，任何 sale_type 擴充實作必須遵守。
規則 / 風險說明
JSON 欄位需後端驗證 option_group_json 格式錯誤會導致前台渲染崩潰。GAS 寫入前須驗證 JSON 可解析、必要欄位齊全；後台編輯框加即時解析提示。
前端價格不可信任 客製化金額由前端計算後送出，但 GAS 確認訂單時必須依商品主檔與選項重新計算，前端傳入的金額僅供顯示，不可直接入帳。
客製化訂單取消 / 改期 / 退款 接單製作商品取消涉及原料損耗，退款政策複雜，需在下單前明確顯示退款說明，並在退款流程中獨立審查，不套用一般現貨退款邏輯。
LINE 通知內容不可過長 客製選項多時，LINE 訊息容易超過限制（5000 字）。訂單 LINE 通知只摘要關鍵資訊，詳細選項導回後台連結查看。
不應直接扣現貨庫存 made_to_order / menu_select 商品通常為接單後製作，不存在現貨庫存邏輯，不應觸發 qty 扣減。若強制扣庫存，庫存數字將失去意義。
原料庫存需另開模組 若未來 menu_select 選項需消耗原料庫存（如：選了「烤魚」扣掉魚片庫存），需獨立建立「庫存配方模組」，不可將原料邏輯混入現有商品庫存欄位，列為 P4 獨立審查。
15

活動報名與預約服務分項規劃

⚠ 本章為規劃草案。P1 現有 event 固定日期活動維持穩定；P2 / P3 / P4 尚未實作，不列入 v1.0 草案封版範圍。

1. activity_type 設計

活動 / 服務主檔新增 activity_type 欄位，以字串識別報名或預約的流程型態。

值說明
event固定日期活動，有名額，可先審核再通知繳費
appointment_service預約服務，客人選時段，一對一或小班制
appointment_event預約制活動 / 客製活動，客人提需求，後台報價
activity_type 與 status 的分工：
status（開放報名 / 名額已滿 / 下架 / 停用）負責活動是否可報名或預約。
activity_type 負責流程型態，決定前台顯示邏輯、後台管理方式與帳務規則。
兩者正交，不可混用。例如：一個「預約服務」可以同時設為「名額已滿」（status=名額已滿，activity_type=appointment_service），代表近期時段已滿，但服務本身仍存在。

2. event：固定日期活動

適用情境：課程、工作坊、營隊、旅療活動、講座、共修活動。

特性說明
固定活動日期活動在特定日期 / 時段舉行，不由客人自選
名額上限住宿 / 不住宿方案各有獨立名額控管
報名審核可設定先審核再通知繳費（requires_review = true）
帳務分流報名費寫入活動帳本，與商品帳本分開
此型態為現有系統已實作的核心模組（活動報名 v26），P1 維持穩定，不做修改。

3. appointment_service：預約服務

適用情境：個案諮詢、健康諮詢、療癒服務、教練課、私人教學、調理服務。

appointment_service 專屬欄位
欄位說明
service_duration_minutes每次服務時長（分鐘），如：60 / 90 / 120
available_days可預約星期幾（如：週二、週四、週六），逗號分隔
available_time_slots可預約時段列表（如：10:00,13:00,15:00），逗號分隔
buffer_minutes每場服務前後預留緩衝時間（分鐘），防止時段重疊
service_location_typeonline：線上；onsite：到府 / 工作室；both：兩者皆可
service_location_note地點說明（如：台北市大安區，或線上 Google Meet）
price_typefixed：固定價格；quote_required：需報價
requires_admin_confirmtrue = 後台需確認預約時段後才成立
cancellation_policy取消政策說明文字，前台顯示（如：需提前 48 小時取消）

前台預約流程

客人選服務頁面 → 查看可用時段 → 選日期 / 時段 → 填報名資料 → 送出預約申請
→ 後台確認時段可用 → LINE 通知客人確認 + 付款資訊
→ 客人付款 → 後台確認收款 → 預約成立，寫入活動帳本

4. appointment_event：預約制活動 / 客製活動

適用情境：私廚體驗、小團體活動、企業包場、家庭日、客製旅療、客製課程。

appointment_event 專屬欄位
欄位說明
min_people最低成行人數
max_people最高接受人數
base_price基本費用（不含人頭費，如場地費）
price_per_person每人費用
available_dates可接受的日期或日期範圍（逗號分隔或規則字串）
quote_requiredtrue = 需後台報價，不直接顯示金額
custom_request_note客製需求說明提示（如：請說明人數、特殊需求、希望日期）
requires_admin_confirmtrue = 後台確認可接案後才成立
deposit_requiredtrue = 需繳訂金
deposit_amount_or_rate訂金金額（固定數字）或比例（如 0.3 = 三成）

前台洽詢流程

客人選活動頁面 → 填日期意向、人數、特殊需求 → 送出洽詢申請（非正式報名）
→ 後台審核可接案 / 報價 → LINE 通知客人報價金額與訂金資訊
→ 客人確認接受 → 繳訂金 → 後台確認訂金收訖 → 正式成立預約
→ 活動結束後繳尾款 → 後台確認尾款 → 寫入活動帳本

6. 前台顯示邏輯

activity_typeCTA 顯示說明
event 「立即報名」 顯示活動日期、名額、費用；進入現有報名流程
appointment_service 「預約服務」 顯示可用時段選擇器；選完填資料；price_type=quote_required 時改顯示「洽詢預約」
appointment_event
quote_required=false		 「預約活動」 填人數 / 日期 / 需求，系統顯示金額估算（base_price + price_per_person × 人數）
appointment_event
quote_required=true		 「洽詢報價」 填人數 / 日期 / 需求送出洽詢，不直接顯示金額，等後台報價後通知

7. 後台管理規劃

activity_type後台管理重點
event 報名名單、審核通過 / 未錄取、確認付款、退款取消、名額統計（現有功能）
appointment_service 依日期 / 時段顯示預約列表；確認 / 拒絕預約；標記付款；時段衝突提示
appointment_event 洽詢列表；填報價金額；確認接案 / 拒絕；標記訂金已付 / 尾款已付；人工修改報價

後台需支援依 activity_type 篩選，避免固定活動、預約服務、客製活動三者混在同一列表難以操作。

8. 帳務規則

情況入帳規則
event 報名費確認付款後寫入活動帳本（現有流程）
appointment_service 服務費確認付款後寫入活動帳本；未來視需求可獨立「服務帳本」
appointment_event 訂金確認訂金收訖後寫入帳本，備註「訂金 — 待尾款」
appointment_event 尾款確認尾款收訖後寫入帳本，備註「尾款結清」；訂金與尾款需分兩筆記錄
quote_required=true 尚未報價不得入帳；進入帳本的前提是報價已確認且款項已收訖

9. 分期建議

階段範圍
P1 現有 event 固定日期活動維持穩定（報名 v26 已封版）。activity_type 欄位預留於活動主檔 schema，不影響現有流程。
P2 activity_type 欄位正式寫入後台活動編輯 UI。現有活動一律標記為 event。後台活動列表加 activity_type 篩選標籤。
P3 appointment_service MVP：前台時段選擇 UI、後台預約確認、LINE 通知客人、付款流程。新增 appointment_db。
P4 appointment_event：洽詢送出、後台報價流程、訂金 / 尾款分段收款、LINE 通知、完整帳務記錄。

10. 安全規則與風險

⚠ 以下規則為強制要求，任何 activity_type 擴充實作必須遵守。
規則 / 風險說明
時段重複預約 appointment_service 同一時段若同時有多人預約，需後端加鎖或先到先得機制。前台需即時更新可用時段，不可依賴快取舊資料。buffer_minutes 需納入衝突判斷。
取消 / 改期 / 退款 每種 activity_type 的取消政策不同（講座 vs 私廚 vs 諮詢），cancellation_policy 需前台顯示，且退款需走退款審查流程，不可自動退款。
訂金與尾款帳務 訂金與尾款必須分兩筆記錄入帳，不可合併為單筆。若退訂金需另建退款帳目，不可直接刪除訂金記錄。
報價金額不可由前端決定 quote_required=true 的服務 / 活動，最終金額由後台人工輸入，前端任何試算僅為估算參考，不可作為實際入帳依據。
LINE 通知內容不可過長 預約確認 LINE 通知需控制字數（建議 500 字以內），詳細預約資訊改以後台連結提供，不全文塞入 LINE 訊息。
個資與健康諮詢內容最小化 預約服務（如健康諮詢、療癒服務）可能涉及健康狀況描述，屬敏感個資，收集欄位需最小化，後台顯示需遮罩，LINE 通知不得包含完整健康資訊（見第 11 章個資處理原則）。
