### 1.3 新增欄位規則：Append-only、預設值、回填方式

所有新增欄位一律遵守以下三條規則，逐表列出：

#### （1）商品主檔 — 新增 5 欄

| 新欄位名稱 | 資料型態 | Append-only | 既有49項商品的預設值回填 |
|-----------|---------|------------|------------------------|
| 供應方 (supplier) | 文字 | 是，加在既有最後一欄之後 | 全部回填為「自營」 |
| 底價 (base_price) | 數字 | 是 | 留空（代表此商品不開放開團，不影響既有銷售） |
| 最低開團售價 (min_group_price) | 數字 | 是 | 留空（同上） |
| 計入團長門檻 (counts_toward_threshold) | 是/否 | 是 | 全部回填為「否」（既有商品預設不計入門檻，需妳日後手動指定） |
| 適用集單模式 (supports_group_gathering) | 是/否 | 是 | 全部回填為「否」（既有商品預設不開放集單，需妳日後手動指定） |

#### （2）進出貨記錄 — 新增 2 欄＋2個異動類型選項

| 單價 (unit_price) | 數字 | 是 | 既有歷史記錄回填為空白或0（不影響既有記錄的既有意義，此欄位僅供本次新增的異動類型使用） |

新增異動類型選項（既有「進貨」「銷售出貨」等既有選項不變，僅新增選項）：

#### （3）客戶訂單 — 新增 4 欄

|-----------|---------|------------|--------------------------|

### 1.4 明確聲明：既有養清倉管主流程不得被破壞

```
本次擴充的唯一前提是：養清倉管既有的商品瀏覽、下單、庫存扣減、進出貨記錄、
帳務記錄、會員積分累計等主流程，在本次擴充完成後，行為必須與擴充前完全一致。

具體驗證方式：
- 一般零售訂單（客戶訂單.團長欄位留空）從下單到扣庫存到寫入商品帳本，
  每一個步驟都必須與擴充前的行為逐位元比對一致，不得有任何差異
- 既有49項商品的既有欄位數值（商品名稱、商品編號、經銷價、零售價、圖片網址、
  說明、分類、狀態、低庫存警示值、建立時間）不得被本次擴充的任何程式碼寫入或覆蓋
- 既有會員的既有欄位（姓名、電話、累積積分等）不得被本次擴充的任何程式碼寫入或覆蓋
- 任何新增的 GAS function，若與既有 function 功能相關，一律用「新建獨立 function」
  的方式撰寫，不得直接修改既有 function 內部邏輯；若必須讓既有 function 感知新欄位
  存在，僅能用「在既有 function 開頭或結尾新增一段判斷分支」的方式處理，該判斷分支
  在新欄位為空/預設值時，必須完全不影響既有 function 原本的輸出結果

若 Codex 複審時發現任何違反上述聲明的實作，一律判定為不通過，退回修正，
不論該次修改在功能面是否完整或正確。
```

## 二、新增表設計（共 5 張新表）

### 2.1 團購商品表

```
欄位：商品ID / 階梯順序 / 門檻數量 / 階梯底價
資料型態：商品ID(文字，對應商品主檔) / 階梯順序(數字，1起) / 門檻數量(數字) / 階梯底價(數字)
預設值：新表無既有資料需回填，全部為本次擴充後新產生的資料
Append-only：本表為全新表，本身即無「既有欄位」概念，未來新增欄位時比照本文件append-only原則辦理
```

- 只有商品主檔「適用集單模式＝是」的商品，才能在此表設定階梯
- 階梯底價由平台統一設定，非團長自訂，階數不限（2階、3階、5階皆可）
- 每一階都要符合「固定加價（設定於團購訂單表）≥ 理貨代收費」

### 2.2 團購訂單表

```
欄位：活動ID / 團長 / 商品ID / 門檻類型 / 固定加價 / 截止時間 / 狀態 / 建立時間
資料型態：活動ID(文字，唯一值) / 團長(文字，可為姐妹團長名字或「幸福緣」) /
        商品ID(文字，對應商品主檔) / 門檻類型(數量/金額) / 固定加價(數字) /
        截止時間(日期時間) / 狀態(集單中/已成團/已下單/流團) / 建立時間(日期時間，系統自動)
```

- 若對應商品在團購商品表已有階梯底價，門檻類型鎖定為「數量」，不可選「金額」
- 「團長」欄位填「幸福緣」時，代表平台自己發起集單（用於平台需要集單湊量拿進貨價的商品）

### 2.3 團購明細表

```
欄位：登記ID / 活動ID / 客人姓名 / 電話 / LINE User ID / 數量 / 登記時間 / 狀態 / 備註
資料型態：登記ID(文字，唯一值) / 活動ID(文字，對應團購訂單表) / 客人姓名(文字) /
        電話(文字) / LINE User ID(文字，經ID Token驗證後取得) / 數量(數字) /
        登記時間(日期時間，系統自動) / 狀態(有效/取消) / 備註(文字，選填)
```

- 同一客人可在不同活動ID下都有登記記錄，互不影響
- 團長確認下單後，此表每筆有效登記各自轉成客戶訂單表一筆獨立訂單

### 2.4 團主/來源表

```
欄位：記錄ID / 會員電話 / 姓名 / 會員等級 / 核准方式 / 核准日期 /
     未取貨次數 / 團購資格狀態 / 來源團長 / 首次帶入時間 / 備註
資料型態：
  記錄ID(文字，唯一值)
  會員電話(文字，對應會員表，作為關聯鍵，不重複存放會員表已有的其他基本資料)
  姓名(文字，僅作核對用途，主要姓名資料仍以會員表為準)
  會員等級(一般會員 / 可申請團長 / 團長，預設「一般會員」)
  核准方式(消費滿NT$10,000限定商品 / 辦理智慧名片付費 / 空白，預設空白)
  核准日期(日期，預設空白)
  未取貨次數(數字，預設0)
  團購資格狀態(正常 / 暫停，預設「正常」)
  來源團長(文字，預設空白，代表一般零售自然加入，非集單帶入)
  首次帶入時間(日期時間，預設空白)
  備註(文字，選填)
```

- 團長資格二擇一（消費滿NT$10,000限定商品／辦理智慧名片付費），皆需人工核准才生效，系統自動判斷達標後僅標記「可申請團長」，不自動開通權限
- 未取貨累計3次自動標記「暫停」，僅影響集單登記資格，不影響一般零售；可隨時由妳人工手動解除
- 來源團長僅作紀錄、不作排他歸屬

### 2.5 團購帳務/分潤表

```
欄位：記錄ID / 訂單ID / 日期 / 角色 / 對象 / 商品ID / 商品名稱 / 數量 / 單價 / 小計金額 / 狀態 / 備註 / 建立時間
資料型態：
  記錄ID(文字，唯一值)
  訂單ID(文字，對應客戶訂單表，同一訂單的多筆拆帳記錄共用同一訂單ID)
  日期(日期)
  角色(供應方 / 團長 / 幸福緣)
  對象(供應方名稱或團長名稱，角色＝幸福緣時可留空或填「幸福緣」本身)
  商品ID(文字，對應商品主檔)
  商品名稱(文字)
  數量(數字)
  單價(數字，該角色實際適用的單位金額，例如供應方角色填底價快照，團長角色填價差)
  小計金額(數字，單價×數量或經計算後的實際金額)
  狀態(正常 / 沖銷，預設「正常」)
  備註(文字，選填)
  建立時間(日期時間，系統自動)
```

- 一筆客戶訂單成立（取貨完成觸發），依三段拆帳邏輯寫入此表3筆或以上記錄（多品項訂單，供應方角色逐項拆分）
- 訂單取消/退貨時，寫入對應沖銷記錄（狀態＝沖銷，金額相等方向相反，訂單ID相同），不刪除原始記錄
- 一般零售訂單（客戶訂單.團長欄位留空）不會在此表產生任何記錄，僅寫入既有商品帳本
- 商品帳本本次除新增「代收代寄服務」這個既有欄位的選項值外，不再涉及團購分潤記錄

## 三、LIFF ID Token 後端驗證流程

### 3.1 完整驗證流程

```
步驟1：客人在LINE內開啟LIFF頁面（會員專區、團長專區、集單登記頁面等）
步驟2：前端呼叫 liff.getIDToken() 取得 ID Token
       （不使用 liff.getProfile() 的結果作為身份依據，因為那是前端自陳，未經驗證）
步驟3：前端呼叫後端 GAS API 時，將 ID Token 作為參數傳送
       （前端不得自行組裝或傳送 line_user_id 欄位給後端）
步驟4：後端 GAS 收到 ID Token 後，呼叫 LINE 官方驗證端點：
       POST https://api.line.me/oauth2/v2.1/verify
       參數：id_token（前端傳來的Token）、client_id（此LIFF對應的Channel ID）
步驟5：LINE 回傳驗證結果，包含欄位 sub（即真正的 LINE User ID）與 exp（有效期限）
步驟6：後端檢查回傳結果是否驗證成功、Token是否在有效期限內
步驟7：驗證通過後，後端才使用這組「驗證過的」User ID，去查詢/寫入
       會員表、團主/來源表、團購明細表等資料
```

### 3.2 不信任前端 line_user_id 的具體規則

```
規則：任何 API 端點，一律不得將前端傳來的「line_user_id」參數直接當作
      查詢或寫入資料的依據，唯一合法的 User ID 來源是步驟4-6驗證後取得的 sub 值

理由：若接受前端直接傳參數，任何人可透過瀏覽器開發工具或攔截封包工具，
      偽造 line_user_id 參數，冒充其他人查詢分潤金額、訂單狀態等隱私資料
```

### 3.3 驗證失敗處理

```
若驗證失敗（Token過期、Token不合法、client_id不符、LINE驗證端點回傳錯誤）：
  → 後端拒絕該次請求，回傳明確錯誤訊息（例如「登入狀態已失效，請重新整理頁面」）
  → 前端顯示錯誤提示，引導客人重新從LINE OA選單或聊天視窗進入
    （重新觸發LIFF初始化以取得新的、有效的ID Token）
  → 嚴禁在驗證失敗時，退而使用前端傳來的電話號碼或其他參數直接查詢資料，
    必須要求重新完成驗證流程
```

## 四、所有寫入點的 LockService 加鎖清單

以下每一個寫入點，涉及「先讀取現有數值 → 判斷或計算 → 寫回」的操作，
一律使用 `LockService.getScriptLock()` 包住整個讀寫區塊，避免並發時互相覆蓋：

```
清單1：團購明細表寫入新登記時，更新該活動的累積數量
      （讀取現有累積值 → 加上新登記數量 → 寫回團購訂單表或即時計算欄位）

清單2：客戶訂單成立時，更新對應商品的「預留庫存」數量
      （讀取現有預留值 → 加上這筆訂單數量 → 寫回）

清單3：訂單狀態改為「已完成」時，將預留庫存轉為正式扣減
      （讀取預留值與實際庫存值 → 計算後寫回實際庫存，並將預留值歸零）

清單4：訂單狀態改為「取消」或「退貨」時，釋放預留庫存
      （讀取預留值 → 扣除該筆訂單數量 → 寫回預留值）

清單5：團長確認下單，將團購明細表的多筆有效登記批次轉成客戶訂單表的多筆訂單
      （需鎖定該活動ID，避免同一活動被重複觸發兩次「確認下單」動作）

清單6：會員表或團主/來源表寫入LINE User ID綁定時
      （讀取是否已有其他記錄使用同一組User ID或同一組電話 → 判斷後寫入，
        避免同時間兩筆綁定請求造成資料衝突）

清單7：未取貨次數累加與團購資格狀態判斷
      （讀取現有未取貨次數 → 加1 → 判斷是否達3次 → 寫回次數與資格狀態，
        避免同一會員的多筆逾期訂單同時觸發判斷造成次數算錯）
```

加鎖原則：每個清單項目的鎖定範圍應盡量精簡（例如鎖定單一活動ID或單一商品ID的範圍，
而非整份表鎖定），避免不必要地拖慢其他不相關操作的執行速度；但寧可鎖定範圍稍大也不可遺漏
任何一個讀寫點。

## 五、底價快照規則

### 5.1 底價的兩種來源

```
簡單開團模式（MVP）：底價來自商品主檔的固定「底價」欄位
集單模式：底價來自團購商品表中，依實際達成數量對應的那一階「階梯底價」
```

### 5.2 快照時機與寫入規則

```
時機：客戶訂單成立當下（不是結算/取貨時才查）
規則：將當時適用的底價（依上述兩種來源判斷後）複製一份數值，
      寫入客戶訂單表.底價快照欄位
效果：之後不管商品主檔的固定底價、或團購商品表的階梯底價之後如何調整，
      這筆訂單永遠依照下單當時寫入的快照數值結算，不會被事後的價格變動影響
```

### 5.3 利潤計算依賴快照，不依賴即時查詢

```
供應方應得金額 = 底價快照 × 數量（逐項計算後加總）
團長應得金額 = Σ（(開團售價 − 底價快照) × 數量） − 理貨代收費
```

任何結算、月結報表、團長專區查詢，計算依據一律是客戶訂單表中已寫入的底價快照欄位，
不得重新查詢商品主檔或團購商品表的「目前」底價。

## 六、沖銷與帳務追溯規則

### 6.1 沖銷觸發時機

```
訂單狀態變更為「取消」或「退貨」時觸發：
- 若該訂單尚未產生任何團購帳務/分潤表記錄（例如客人尚未取貨、根本還沒結算過），
  則不需要沖銷，因為原本就還沒記過帳，直接將訂單標記為取消/退貨即可
- 若該訂單已經產生過分潤記錄（已完成後才發現要退），才需要寫入沖銷記錄
```

### 6.2 沖銷記錄的寫入方式

```
針對原始記錄的訂單ID，寫入對應的沖銷記錄：
  訂單ID：與原始記錄相同
  角色：與原始記錄相同（供應方/團長/幸福緣，逐筆對應）
  金額：與原始記錄相等，但正負號相反
  狀態：標記為「沖銷」
不得直接刪除原始記錄，確保異動軌跡完整可查
```

### 6.3 帳務可追溯原則

```
任何一筆金額，都必須能夠回答以下三個問題：
1. 這筆錢屬於哪一筆訂單（訂單ID）
2. 這筆錢屬於誰（角色、對象）
3. 這筆錢是怎麼計算出來的（對應的商品、數量、單價/底價快照）

月結報表、團長專區、供應方對帳單，皆從團購帳務/分潤表篩選＋加總產生，
篩選時必須將「正常」與「沖銷」狀態的記錄一併納入計算（沖銷記錄會自動抵銷對應的正常記錄），
不得只呈現「正常」狀態記錄而忽略沖銷記錄，否則會高估應付金額
```

## 七、庫存預留與釋放規則

### 7.1 三個時點

```
時點1（訂單成立）：
  客戶訂單寫入，狀態＝「待取貨」
  → 對應商品的預留庫存數量增加該筆訂單的數量
  → 實際庫存數字不變動

時點2（取貨完成）：
  訂單狀態由「待取貨」改為「已完成」
  → 該筆訂單的預留數量轉為正式扣減
  → 實際庫存數字在此刻才真正減少
  → 預留庫存數量歸零（該筆訂單部分）

時點3（取消或退貨）：
  訂單狀態改為「取消」或「退貨」
  → 該筆訂單的預留數量歸零釋放，回到可販售庫存
  → 若訂單先前已進入「已完成」狀態才轉為退貨，需視實際退貨狀況判斷
    是否將貨品實際入庫（回補實際庫存），此判斷由妳人工於後台操作確認
```

### 7.2 可販售庫存計算公式

```
可販售庫存 = 商品主檔實際庫存 − 所有「待取貨」狀態訂單的預留數量總和
```

### 7.3 超賣防範

```
訂單成立時，系統應檢查「可販售庫存」是否足夠這筆訂單的數量：
- 足夠：正常成立，進入時點1的預留流程
- 不足：後台應提示警告，是否強制阻擋依商品性質由妳決定
此檢查與預留數量的讀寫，皆須包在第四章清單2的LockService鎖定範圍內，
確保多個團長同時賣同一商品時，數字不會因並發而算錯
```

## 八、團主／團員 UI 流程

### 8.1 團主（團長）開團流程

```
第一階段（MVP）：
  團長私下告知妳想開的商品、固定加價、期限
  → 妳在後台建立團購訂單表記錄
  → 系統產生專屬集單連結（例如 ?campaign=A001）
  → 團長自行分享推廣，自己盯進度，自己決定何時確認下單

第二階段（進階，待第一階段驗證跑順後啟動）：
  團長透過LIFF自助登入
  → ID Token驗證後，確認會員等級為「團長」
  → 自行挑選已標記「適用集單模式」的商品
  → 設定固定加價、門檻類型（已有團購商品表階梯底價者鎖定為「數量」）、截止時間
  → 送出建立，系統自動產生活動ID與集單連結
```

### 8.2 團員（客人）下單流程

```
客人點擊團長分享連結，或訂購頁面「團購中」分類頁籤的活動卡片
→ 進入LIFF集單頁面，顯示商品資訊、階梯進度、剩餘時間
→ 頁面自動觸發LIFF取得ID Token（客人無感，不需手動操作）
→ 客人用「+/−」按鈕輸入登記數量（非自由輸入框，避免亂填）
→ 送出登記
→ 後端驗證ID Token → 確認為合法使用者 → 寫入團購明細表（第四章清單1鎖定範圍內）
→ 回傳登記成功確認，並觸發LINE推播通知（登記成功、進度更新等情境）
```

### 8.3 團長專區（自助查詢）

```
團長點擊「團長專區」入口
→ LIFF自動取得ID Token → 後端驗證 → 確認會員等級為「團長」
→ 是團長：顯示her自己開過的所有活動列表（進行中／已結束）、
  每團底下每筆訂單的取貨狀態、累積分潤金額（可依月份篩選，
  數字來源皆為團購帳務/分潤表，見第六章追溯原則）
→ 不是團長：顯示提示訊息，導回一般會員專區，不開放進入
→ 看不到：底價、階梯底價明細、供應方名稱、其他團長的資料
  （隱私邊界與既有規則一致，不因自助查詢功能而放寬）
```

### 8.4 會員專區（一般客人／首次辨識）

```
客人點擊「會員專區」入口
→ LIFF自動取得ID Token → 後端驗證取得真正的LINE User ID
→ 查詢會員表：此User ID是否已有記錄
  → 有記錄：直接進入會員專區，顯示訂單、點數、優惠，不需再輸入手機號碼
  → 無記錄：顯示「首次填寫資料」表單
    → 若輸入電話號碼在會員表已有既有資料：比對後直接補綁LINE User ID，
      不需重填姓名等其他既有資料（此寫入動作納入第四章清單6鎖定範圍）
    → 若電話號碼在會員表也查無資料：視為全新會員，需完整填寫姓名、電話，
      建立新會員記錄並同時綁定LINE User ID
→ 綁定完成後，直接導向會員專區，不需再次操作
```

### 8.5 後台查單與匯出統計

```
後台查單：
  以電話號碼查詢客人所有「待取貨」訂單（可能橫跨不同團長的不同活動）
  → 勾選欲結清的訂單（可複選）→ 逐筆確認商品/數量/收款方式/金額 → 填寫備註
  → 確認取貨 → 系統對每筆勾選的訂單各自獨立觸發：
    第五章底價快照結算、第二章2.5團購帳務/分潤表寫入、第七章時點2庫存扣減、
    訂單狀態改為「已完成」（多筆訂單各自獨立處理，不合併計算）

匯出統計：
  月結報表：依「角色＋對象」篩選團購帳務/分潤表，產出供應方／團長／幸福緣
    各自的逐筆明細單（日期、訂單ID、商品、數量、單價、小計、加總），可匯出/列印
  團長專區查詢：即時篩選團購帳務/分潤表中「角色＝團長且對象＝該團長」的記錄，
    供團長自助查詢，數字與月結報表一致
```
